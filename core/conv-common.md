# 개발 컨벤션 — 공통

> 모든 스택에서 적용되는 공통 규칙.
>
> CRITICAL: 코드 수정 전 반드시 해당 파일을 먼저 읽을 것. 기억에 의존한 수정 금지.

## 공통

| 항목 | 규칙 |
|------|------|
| 인코딩 | UTF-8 |
| 개행 | LF (Unix) |
| 파일 끝 | 빈 줄 1개로 종료 |
| 주석 언어 | 한국어 허용 (비즈니스 로직), 변수/함수명은 영어 |
| TODO 표기 | `// TODO(#{이슈번호}): 내용` |
| 매직넘버 금지 | 상수 또는 enum으로 정의 |
| 로그 레벨 | ERROR(장애), WARN(예외상황), INFO(주요흐름), DEBUG(개발용) |
| 민감정보 | 개인정보(전화번호, 이메일 등) 로그/응답 마스킹 |

## 코딩 원칙

| 원칙 | 설명 |
|------|------|
| 명시성 > 영리함 | 동료가 빠르게 이해할 수 있는 코드를 쓴다 |
| 단일 책임 | 함수/클래스는 하나의 책임에 집중. 함수는 최대 50줄, 파라미터는 최대 5개 |
| 부작용 금지 | 숨겨진 부작용을 만들지 않는다, 상태 변경은 명시적으로 |
| 정확성 > 성능 | 정확성을 먼저 확보하고 성능 최적화 |
| 보수적 구현 | 요구사항이 애매하면 확장하기 쉬운 보수적 구현 선택 |
| 주석은 why | "무엇"이 아니라 "왜"를 설명. 주석은 전체 코드 라인의 20%를 넘기지 않음 |
| 의존성 신중 | 구체적 이유 없이 새 의존성 도입 금지 |
| 조기 추상화 금지 | 비슷한 코드 3줄이 성급한 추상화 1개보다 낫다 |

### 해야 할 것 vs 하지 말아야 할 것

해야 할 것:
- 에러 메시지는 한국어로, 원인과 해결 방법을 함께 포함
- 수정 전 해당 코드의 호출부를 반드시 확인
- 변경 사유를 산출물에 기록

하지 말아야 할 것:
- "나중에 쓸 수 있으니까" 사용되지 않는 코드를 남기지 말 것
- 한 PR에 3개 이상의 관심사를 섞지 말 것
- 빈 catch 블록으로 에러를 무시하지 말 것
- 요청 범위 밖의 리팩토링/개선을 임의로 하지 말 것

## 코드 변경 원칙

| 항목 | 규칙 |
|------|------|
| 기존 코드 존중 | 주변 코드 패턴과 일관성 유지 |
| 최소 변경 | 요청된 범위만 수정, 불필요한 리팩토링 자제 |
| 영향 범위 확인 | 수정 전 해당 메서드/컴포넌트 호출부 확인 |
| 하위 호환 | API 변경 시 기존 클라이언트 호환 고려 |
| 변경 사유 기록 | development.md에 무엇을/왜/남은 리스크 기록 |

## 로깅 / 관측성

| 항목 | 규칙 |
|------|------|
| 비즈니스 마일스톤 | 주요 이벤트(생성, 수정, 삭제 등)는 INFO 로깅 |
| 구조화 로깅 | 파라미터 바인딩 사용 (f-string에 직접 넣지 않음) |
| 노이즈 제거 | 정상 흐름 반복 DEBUG는 운영에서 제거 |
| 운영자 vs 사용자 | 로그는 상세히, 에러 응답은 안전하게 |

## 테스트 원칙

| 항목 | 규칙 |
|------|------|
| 테스트 필수 | 의미 있는 로직 변경에는 테스트 동반 (불가 시 사유 기록) |
| 빠르고 결정적 | 외부 의존 없이 빠르게, 매번 같은 결과 |
| 구현 과적합 지양 | 내부 구현에 결합된 깨지기 쉬운 테스트 피하기 |
| 의미 있는 커버리지 | 수치보다 핵심 분기/에러 경로 우선 |
