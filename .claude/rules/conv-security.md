# 개발 컨벤션 — 보안 코딩 원칙

> 모든 스택에서 적용되는 보안 규칙.
>
> CRITICAL: 금지 패턴(SEC-AP-001~008) 위반 코드를 절대 생성하지 말 것.

## 입력값 검증

| 항목 | 규칙 |
|------|------|
| SQL Injection | ORM 파라미터 바인딩 필수, raw SQL 문자열 연결 금지 |
| XSS | 입력값 이스케이프, `dangerouslySetInnerHTML` 사용 시 DOMPurify 적용 |
| 파일 업로드 | 확장자 화이트리스트 + MIME 타입 검증 + 파일 크기 제한 |
| URL 리다이렉트 | 허용 도메인 화이트리스트 검증 |
| 프론트 검증 | UX 목적만, 최종 검증은 서버 사이드 필수 |

## 인증 / 인가

| 항목 | 규칙 |
|------|------|
| API 인증 | 인증 미들웨어/데코레이터 적용, 제외 엔드포인트는 명시적 화이트리스트 |
| 권한 검증 | 서버 사이드 필수 — 프론트 UI 숨김만으로 접근 통제 금지 |
| 비밀번호 저장 | bcrypt 단방향 해시, 평문 저장/로깅 금지 |
| JWT | 적절한 만료 시간, refresh token 회전 |

## 암호화

| 항목 | 규칙 |
|------|------|
| 통신 | HTTPS (TLS 1.2+) |
| 금지 알고리즘 | MD5, SHA-1, DES 사용 금지 |
| 키 관리 | 소스코드에 암호키 하드코딩 금지, 환경변수 사용 |

## 정보 노출 방지

| 항목 | 규칙 |
|------|------|
| 에러 응답 | 사용자에게 일반 메시지만, 스택 트레이스는 서버 로그에만 |
| 디버그 코드 | 운영 배포 전 제거 |
| CORS | 허용 도메인 명시적 지정 |
| 소스맵 | 프로덕션 빌드 시 비공개 |
| .env | Git에 커밋 금지 (.gitignore 필수) |

## 금지 패턴 (SEC-AP)

| ID | 금지 패턴 | 올바른 방식 |
|----|----------|------------|
| SEC-AP-001 | SQL 문자열 연결 | ORM 파라미터 바인딩 |
| SEC-AP-002 | 하드코딩 비밀정보 | 환경변수 |
| SEC-AP-003 | 취약 암호화 (MD5, SHA-1) | SHA-256, AES-256 이상 |
| SEC-AP-004 | 비밀번호 평문 저장/로깅 | bcrypt 해시 + 로그 마스킹 |
| SEC-AP-005 | 프론트만 의존하는 권한 검증 | 서버 사이드 권한 검증 |
| SEC-AP-006 | 클라이언트에 스택 트레이스 노출 | 일반 에러 메시지 |
| SEC-AP-007 | dangerouslySetInnerHTML 무방비 사용 | DOMPurify 적용 |
| SEC-AP-008 | localStorage에 민감정보 저장 | httpOnly 쿠키 또는 메모리 |

REMINDER: SEC-AP 금지 패턴 위반 코드를 절대 생성하지 말 것. 텍스트 설명으로든, 코드 주석으로든, 예시 코드로든 금지 패턴을 사용하는 코드를 작성하지 않는다.
