# /ace audit — 코드베이스 진단

## 사용법

```
/ace-audit              전체 프로젝트 진단
/ace-audit src/         특정 디렉토리만 진단
```

## 목적
기존 코드베이스를 **읽기 전용**으로 분석하여 구조적 문제, 품질 이슈, 보안 취약점, 성능 병목을 진단한다.
코드를 수정하지 않으며, 발견 사항을 심각도별로 분류하여 보고한다.

## 제약 사항
- **읽기 전용**: 코드 수정, 파일 생성(audit.md 제외), 패키지 설치 금지
- 태스크/이슈번호 불필요
- 추측하지 말 것 — 코드에서 직접 확인한 사실만 보고

## ⚡ 이어하기 규칙 (필수)

### 실행 전 체크
1. `workspace/audit.md` 파일이 이미 존재하는지 확인한다.
2. 존재하면 frontmatter의 `status`를 읽는다.
   - `status: done` → "이미 완료된 진단입니다. 재진단하시겠습니까?" 확인
   - `status: in_progress` → **이어하기 모드** 진입

### 이어하기 모드
1. 기존 audit.md를 읽어 **어떤 섹션까지 완료되었는지** 파악한다.
2. 미완료 섹션부터 이어서 분석한다.
3. 이미 완료된 섹션은 **절대 다시 작성하지 않는다.**

### 섹션별 저장
각 섹션 완료 시마다 audit.md를 **즉시 저장**한다.

## 에이전트 사용 규칙

### 에이전트 호출 조건
- **메인 세션이 직접 수행** (기본): 대부분의 감사 작업
- **에이전트 위임**: 분석 대상 파일이 30개 이상인 대규모 코드베이스 감사만
- **병렬 에이전트 금지**: 같은 산출물을 여러 에이전트가 동시에 쓰지 않는다
- **큰 파일 주의**: 필요한 부분만 offset/limit로 읽어서 요약 후 전달

### 폴백
1. 에이전트가 타임아웃/에러/불완전한 결과를 반환하면 메인 세션이 직접 수행한다.
2. `<!-- 에이전트 실패 — 메인 세션에서 직접 수행 -->` 주석을 남긴다.
3. 에이전트를 재시도하지 않는다.
4. 에이전트의 중간 출력을 직접 읽지 않는다. 최종 결과만 수신한다.

## 실행 흐름

### Step 1: 프로젝트 스캔

프로젝트의 전체 구조를 파악한다.

1. **디렉토리 구조** — 주요 디렉토리와 파일 분포
2. **기술 스택 식별** — package.json, requirements.txt, Dockerfile 등에서 추론
3. **규모 파악** — 파일 수, 총 라인 수, 언어별 비율
4. **진입점 파악** — 앱 시작점, 라우팅 구조, 설정 파일

→ audit.md에 "프로젝트 개요" 섹션 저장

### Step 2: 구조 분석

아키텍처와 코드 구조를 진단한다.

**체크 항목:**
- 디렉토리 구조가 일관된 패턴을 따르는가
- 관심사 분리가 되어 있는가 (레이어, 모듈)
- 순환 의존성이 있는가
- 300줄 초과 파일 목록
- 단일 책임 원칙 위반 (하나의 파일이 너무 많은 역할)
- 미사용 파일/코드 (import 없는 모듈, dead code)
- 설정/환경 관리 방식

→ audit.md에 "구조 분석" 섹션 저장

### Step 3: 코드 품질

코드 품질과 유지보수성을 진단한다.

**체크 항목:**
- 네이밍 일관성 (camelCase/snake_case 혼용 등)
- 에러 처리 패턴 (빈 catch, 에러 무시, 일관성)
- 코드 중복 (유사 로직 반복)
- 타입 안전성 (any 남용, 타입 누락)
- 테스트 현황 (커버리지, 테스트 유무, 테스트 품질)
- 하드코딩된 값 (매직넘버, 문자열)
- 주석 상태 (과다/부족, 오래된 주석)

→ audit.md에 "코드 품질" 섹션 저장

### Step 4: 보안 점검

보안 취약점을 진단한다. SEC-AP 금지 패턴을 기준으로 검사한다.

**체크 항목:**
- SEC-AP-001: SQL 문자열 연결
- SEC-AP-002: 하드코딩 비밀정보 (.env, 코드 내 API 키)
- SEC-AP-003: 취약 암호화 (MD5, SHA-1)
- SEC-AP-004: 비밀번호 평문 저장/로깅
- SEC-AP-005: 프론트만 의존하는 권한 검증
- SEC-AP-006: 클라이언트에 스택 트레이스 노출
- SEC-AP-007: dangerouslySetInnerHTML 무방비 사용
- SEC-AP-008: localStorage에 민감정보 저장
- 추가: CORS 설정, 인증/인가 미들웨어, 입력값 검증

→ audit.md에 "보안 점검" 섹션 저장

### Step 5: 성능 및 의존성

성능 병목과 의존성 문제를 진단한다.

**체크 항목:**
- N+1 쿼리 패턴
- 불필요한 리렌더링 (React: useEffect 의존성, 메모이제이션 누락)
- 번들 크기 이슈 (불필요한 대형 패키지)
- 의존성 상태 (outdated, deprecated, 보안 취약점)
- 캐싱 전략 유무
- API 호출 패턴 (워터폴, 중복 호출)

→ audit.md에 "성능 및 의존성" 섹션 저장

### Step 6: 종합 보고

모든 발견 사항을 심각도별로 정리한다.

**심각도 분류:**
- CRITICAL: 즉시 수정 필요 (보안 취약점, 데이터 유실 위험)
- HIGH: 빠른 수정 권장 (심각한 품질 문제, 성능 병목)
- MEDIUM: 개선 권장 (구조 문제, 유지보수성)
- LOW: 선택적 개선 (스타일, 사소한 개선)

**종합 점수 (각 10점 만점):**
- 구조: X/10
- 품질: X/10
- 보안: X/10
- 성능: X/10
- 종합: X/10

→ audit.md에 "종합 보고" 섹션 저장, status: done

## 출력 파일

`workspace/audit.md` — frontmatter:

```yaml
---
status: in_progress  # → done
date: {{date}}
target: {{분석 대상 경로}}
summary: "종합 X/10 — CRITICAL X건, HIGH X건"
---
```
