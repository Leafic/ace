# /ace analyze — 요구사항 분석

## 사용법

```
/ace analyze [이슈번호]
```

## 선행 조건
- 태스크가 생성되어 있어야 함 (`/ace task start`)

## 실행 흐름

### Step 1: 입력 수집

분석 대상 자료를 수집한다:
- 이슈 트래커의 설명/첨부
- `workspace/tasks/{번호}/planning/` 폴더의 기획서
- 사용자가 대화로 제공한 요구사항

### Step 2: 에이전트 호출

`analyzer` 에이전트를 호출하여 다관점 분석을 수행한다.

**에이전트 입력:**
```
- 이슈번호, 태스크 폴더 경로
- 기획 자료 (변환된 마크다운)
- 기존 analysis.md (재분석 시)
- 프로젝트 도메인 용어집 (있으면)
```

### Step 3: 산출물 작성

`workspace/tasks/{번호}/analysis.md`를 작성한다.

**분석 관점 (solo 모드 — 간소화):**
1. **기능 요구사항**: 구현해야 할 기능 목록
2. **데이터 모델**: 필요한 엔티티/관계
3. **API 설계 힌트**: 필요한 엔드포인트 윤곽
4. **예외/엣지케이스**: 고려할 예외 상황
5. **미해결 사항**: 기획 모호성, 추가 확인 필요 항목

**분석 관점 (team 모드 — 상세):**
1. **화면 목록**: 영향받는 화면 식별
2. **화면별 분석**: UI/데이터/FE/BE/예외 관점
3. **공통 요구사항**: 인증, 권한, 검증 등
4. **레거시 영향**: 기존 시스템 영향 분석
5. **비기능 요구사항**: 성능, 보안, 접근성
6. **미해결 사항**: 추가 확인 필요 항목

### Step 4: 상태 갱신

- analysis.md frontmatter: `status: done`
- taskDetail.json: `steps.analysis.status: completed`

## 재분석 (Incremental)

이미 analysis.md가 존재하면:
1. 변경된 입력 자료만 식별
2. 기존 분석 유지 + 변경 부분만 업데이트
3. rev 번호 증가 + 변경 이력 기록

## 완료 시 출력

```
═══ 분석 완료 요약 (#{번호}) ═══

[핵심 결론]
- (1-3줄)

[주요 결정 사항]
1. ...

[미해결/리스크]
- ...

[다음 단계 참고]
- 설계 시 확인할 핵심 포인트
```
